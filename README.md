# Code Introduction

- config_loader.py: Parses Common.cfg and PeerInfo.cfg, and computes relevant constants such as numPieces = FileSize/PieceSize
- connection.py: class that handles starting connections, as well as sending and receiving messages 
- peer_network.py: Handles networks of connections with multiple peers, sending messages between them, and reconnecting/stopping connections
- protocol.py: Contains the MessageFramer which safely buffers real-time raw TCP streams into complete structured messages and unpacks the 32-byte Handshake.
- state.py: Thread-safe state store containing Bitfield structures and tracks piece inventory and choke/interest status for the local peer and all active neighbors.

# P2P Project Walkthrough

This section maps our implementation directly to the core project requirements.

### 1. Project Setup and Configuration Files
**Requirement:** The system relies on two configuration files: `Common.cfg` for global properties (UnchokingInterval, NumberOfPreferredNeighbors, FileSize, etc.) and `PeerInfo.cfg` for tracking peers, their listening ports, and an initial flag indicating whether they possess the complete file.

**Implementation:**
- **[config_loader.py](./config_loader.py):** Parses `Common.cfg` to calculate constants such as `numPieces = ceil(FileSize / PieceSize)`. It also extracts host, port, and file status details from `PeerInfo.cfg` into a list of `PeerInfo` objects.
- **[main.py](./main.py):** The main entrypoint. It reads the peer ID from the command line, loads configurations via the config loader, and initializes the `PeerNetwork`.

### 2. File and Directory Management
**Requirement:** Each peer must isolate complete files and partial pieces within its own subdirectory (e.g., `peer_[peerID]/`).

**Implementation:**
- **[file_manager.py](./file_manager.py):** The `FileManager` class handles checking, reading, and writing file pieces. It stores pieces under `mem/peer_{peer_id}/pieces/`. Once all pieces are downloaded, it calls `assemble_file()` to merge the binaries into a final file under the base `mem/peer_{peer_id}` directory.

### 3. TCP Connections and Startup
**Requirement:** A newly started peer listens on its designated port, but only initiates TCP connections to peers that were started before it. 

**Implementation:**
- **[peer_network.py](./peer_network.py):** Calling `PeerNetwork.start()` initiates an `_accept_loop` thread that listens for incoming connections from newer peers. Concurrently, `_connect_to_previous_peers()` iterates down the peer list and establishes socket connections to peers initialized earlier.
- **[connection.py](./connection.py):** A lightweight, thread-safe wrapper around TCP sockets. It runs a background `_recv_loop` that continuously pulls data from the TCP stream and passes it to message handlers.

### 4. Handshakes and Bitfields
**Requirement:** Upon connection establishment, both peers immediately transmit a 32-byte handshake: `P2PFILESHARINGPROJ` + 10 zero bytes + the 4-byte Peer ID. Any peer with existing pieces must follow up immediately with a bitfield message.

**Implementation:**
- **Handshaking:** As soon as sockets connect in `peer_network.py`, the payload generated by `create_handshake` in `protocol.py` is transmitted across the socket.
- **Message Parsing:** The `MessageFramer` in **[protocol.py](./protocol.py)** buffers the incoming bytes. It decodes the initial 32 bytes for the handshake and parses subsequent payload data using their 4-byte length prefixes.
- **Bitfield:** After the handshake is completed, the implementation checks if the peer possesses any pieces (`self.peer_state.my_bitfield.count() > 0`) and sends out a `BITFIELD` message if true.

### 5. Peer State & Interest Operations
**Requirement:** When a peer receives a `BITFIELD` or a `HAVE` message revealing piece indices it lacks, it must identify the deficit and reply with `INTERESTED` or `NOT_INTERESTED`.

**Implementation:** 
- **[state.py](./state.py):** Acts as the centralized state store (`PeerState`), tracking piece inventory for the local node and neighboring peers. Whenever state updates occur, `PeerState.get_interesting_pieces(remote_id)` determines if any missing pieces are held by a neighbor.
- **Interest Processing:** In `PeerNetwork._sync_interest_for_peer()`, the system calculates any offset between neighbor holdings and local requirements to selectively send `INTERESTED` or `NOT_INTERESTED` messages.

### 6. Choking and Unchoking
**Requirement:** 
1. Every `p` seconds, determine up to `k` preferred neighbors based on download rates (or randomly, if the peer already has the complete file). Choke all others.
2. Every `m` seconds, randomly select an optimistically unchoked neighbor from the pool of choked but interested peers.

**Implementation:**
- **Timers:** Both `_unchoking_timer_loop` and `_optimistic_timer_loop` execute persistently on their respective interval bounds.
- **Preferred Selection (`_select_preferred_neighbors`):** The system reliably polls accrued download sizes from `PeerState`. If the complete file is present, it chooses randomly. Otherwise, neighbors are sorted by download rate to retrieve the top `k`, and updated state causes the transmission of `UNCHOKE` or `CHOKE` messages.
- **Optimistic Selection (`_select_optimistic_neighbor`):** Randomly selects one neighbor from the set of interested, choked peers, transmits an `UNCHOKE` to them, and transitions the prior optimistic peer back to a `CHOKE` state.

### 7. Requesting Pieces
**Requirement:** Upon receiving an `UNCHOKE`, request one missing piece at random. Request pipelining is disabled: the client must wait for the full `PIECE` before dispatching a subsequent `REQUEST`.

**Implementation:**
- Receiving an `UNCHOKE` triggers `_maybe_request_piece` inside **[peer_network.py](./peer_network.py)**. It uses `state.select_random_piece(remote_id)` to choose an index and dispatches the `REQUEST`.
- When the `PIECE` payload arrives, `FileManager.write_piece` flushes the chunk to disk. Upon successful verification, the system immediately continues the cycle by calling `_request_next_piece(remote_id)`.
- Reaching a newly downloaded state concurrently broadcasts a `HAVE` message to all neighbors.

### 8. Logging
**Requirement:** Essential events such as TCP connection metrics, preferred neighbor selections, piece downloads, optimistic unchokes, etc., must be logged precisely and mapped locally per individual peer.

**Implementation:**
- **[logger.py](./logger.py):** Exposes formatting methods (e.g., `log_connection_made`, `log_downloaded_piece`) fulfilling the specified format requirements.
- **Integrations:** `PeerNetwork` emits these logs during key state transitions, persisting cleanly into an isolated file (`logs/log_peer_[ID].log`).

### 9. Knowing When to Stop
**Requirement:** The peer process must automatically and gracefully terminate upon realizing that all peers in the cluster contain the complete file.

**Implementation:**
- `PeerNetwork._check_termination()` handles this termination logic. 
- Incoming `HAVE` or `BITFIELD` broadcasts constantly update the status of neighbors, allowing `PeerState.all_neighbors_complete(expected_neighbors)` to flip true once total network completion is reached.
- Once total replication equals true, the root thread wakes up, yields briefly for remaining trailing TCP broadcasts to flush, and cleanly exits via `os._exit(0)`.
