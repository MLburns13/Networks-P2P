# P2P Project Walkthrough

Here is a step-by-step walkthrough of the project, matching the requirements outlined in `project description-v1.pdf` exactly with the implementation in your codebase.

## 1. Project Setup and Configuration Files
**PDF Requirement:** The system relies on two configuration files: `Common.cfg` for global properties (UnchokingInterval, NumberOfPreferredNeighbors, FileSize, etc.) and `PeerInfo.cfg` for knowing about other peers, their listening ports, and whether they start with the complete file.

**Code Match:**
- **[config_loader.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/config_loader.py):** Implements `parse_common_cfg` to parse `Common.cfg` into a dictionary, automatically calculating `numPieces = ceil(FileSize / PieceSize)`. It implements `parse_peerinfo_cfg` to retrieve the details (host, port, has_file flag) of every peer into a list of `PeerInfo` objects.
- **[main.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/main.py):** Acts as the entrypoint `peerProcess`. It reads the peer ID from the command-line argument, loads configurations using `config_loader.py`, and initializes the `PeerNetwork`.

## 2. File and Directory Management
**PDF Requirement:** Each peer must isolate its complete files or partial pieces inside its own subdirectory `peer_[peerID]/`.

**Code Match:**
- **[file_manager.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/file_manager.py):** The `FileManager` class handles checking, reading, and writing pieces to the disk. It safely isolates storage under `mem/peer_{peer_id}/pieces/`. When a file is completed, it calls `assemble_file()` to stitch all the binary piece blobs into the final `FileName` in the base `mem/peer_{peer_id}` directory.

## 3. TCP Connections and Startup State
**PDF Requirement:** A newly started peer listens on its port. It then proactively makes TCP connections ONLY to peers that started before it (as ordered in `PeerInfo.cfg`).

**Code Match:**
- **[peer_network.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/peer_network.py):** The `PeerNetwork.start()` method initiates an `_accept_loop` thread for listening to peers starting *after* it. It also spawns a `_connect_to_previous_peers()` thread that loops over the `peers` list, establishing connections to any peer whose order precedes its own `my_index`.
- **[connection.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/connection.py):** A lightweight thread-safe wrapper around `socket` instances. It creates a background `_recv_loop` that continuously pulls bytes from the TCP socket and triggers the `on_bytes_handler` callback.

## 4. Handshake and Bitfields
**PDF Requirement:** On connection establishment, both sides immediately send a 32-byte handshake: `P2PFILESHARINGPROJ` + 10 zero bytes + 4-byte Peer ID. Following that, peers who have any pieces immediately send a bitfield message.

**Code Match:**
- **Handshaking:** In `peer_network.py`, both the `_accept_loop` and `_connect_to_previous_peers` immediately send the handshake generated by `protocol.py:create_handshake` as soon as the socket connects. 
- **Message Parsing:** Bytes are fed to `MessageFramer` in **[protocol.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/protocol.py)**. The framer accurately unpacks the initial 32 bytes to extract the `remote_id`, and then unpacks subsequent messages based on the 4-byte length prefix.
- **Bitfield:** Once the handshake is parsed inside the `on_bytes_handler` for `handshake`, the code checks if `self.peer_state.my_bitfield.count() > 0`; if so, it sends the `BITFIELD` message.

## 5. Peer State & Interest Operations
**PDF Requirement:** When a peer receives a `BITFIELD` or `HAVE` message revealing pieces it doesnâ€™t have, it must identify the deficit and reply with `INTERESTED` or `NOT_INTERESTED`.

**Code Match:** 
- **[state.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/state.py):** Stores the centralized brain for each peer's internal state mechanism via `PeerState` and `Bitfield`. It remembers which neighbor has which piece. When an incoming message alters a neighbor's bitfield, `PeerState.get_interesting_pieces(remote_id)` gets computed.
- **Interest Processing:** Found inside `PeerNetwork._sync_interest_for_peer()`. It compares what `PeerState` knows the neighbor has with what we need, conditionally mapping to sending `INTERESTED` or `NOT_INTERESTED` packets over the TCP socket.

## 6. Choking/Unchoking Algorithm
**PDF Requirement:** 
1. Every `p` seconds, determine up to `k` preferred neighbors based on download rates (or randomly, if we have the complete file). Choke everyone else.
2. Every `m` seconds, randomly pick an optimistically unchoked neighbor from choked but interested peers.

**Code Match:**
- **Timer Threads:** Spawned inside `PeerNetwork.start()`: `_unchoking_timer_loop` and `_optimistic_timer_loop` constantly trigger on the bounds of `UnchokingInterval` and `OptimisticUnchokingInterval`.
- **Preferred Selection (`_select_preferred_neighbors`):** Retrieves the payload sizes accrued securely from `PeerState.get_and_reset_download_rates()`. If `has_complete_file()` is truthy, it picks random neighbors. Otherwise, it orders neighbors by download bytes, selects the top `k`, updates states, and sends the `UNCHOKE`/`CHOKE` network payloads on changes.
- **Optimistic Selection (`_select_optimistic_neighbor`):** Randomly selects one neighbor via `self.peer_state.get_choked_interested_neighbors()`, sends an `UNCHOKE` specific to them, and cleanly ensures the previous optimistic peer goes back to a `CHOKE` state.

## 7. Requesting Pieces and PIPELINING
**PDF Requirement:** When an `UNCHOKE` is received, we request one randomly un-requested piece. We do *not* pipelineâ€”we wait until we receive the `PIECE` before dispatching the next `REQUEST` for that specific peer.

**Code Match:**
- When receiving `UNCHOKE` inside `peer_network.py`'s `on_bytes_handler`, it fires `_maybe_request_piece`. This leverages `state.select_random_piece(remote_id)` (obeying random selection rule) and sends `REQUEST`.
- When receiving `PIECE`, `_handle_piece_received` processes it and saves it via `FileManager.write_piece`. After it safely writes, logs, and increments data structures, it checks for completion and triggers `_request_next_piece(remote_id)`, creating a neat, pipelineless request loop.
- It concurrently calls `_broadcast_have(piece_index)`, keeping adjacent peers notified of our newly retrieved fragment.

## 8. Logging
**PDF Requirement:** Required system logs such as `TCP connection`, `preferred neighbors`, `downloading a piece`, `optimistically unchoked neighbor`, choking/unchoking, etc., must be logged precisely and uniquely per peer.

**Code Match:**
- **[logger.py](file:///Users/xuhanghe/Desktop/UF/CNT5106C%20Computer%20Networks/Networks-P2P/logger.py):** Contains standard methods matching every requested PDF logging format. E.g., `log_connection_made`, `log_preferred_neighbors`, `log_downloaded_piece`. 
- **Integrations:** `PeerNetwork` relies on `self.logger.log_...` inline everywhere an event occurs. All logs fall cleanly within `logs/log_peer_[ID].log`.

## 9. Global Termination Condition
**PDF Requirement:** The peer process stops gracefully once *not just itself*, but *all peers in the cluster* have the complete file.

**Code Match:**
- Handled efficiently by `PeerNetwork._check_termination()`. 
- Since we continuously update knowledge of neighbors when we receive `HAVE` or `BITFIELD`, `PeerState.all_neighbors_complete(expected_neighbors)` eventually turns true when all nodes own exactly $n$ pieces.
- When `_check_termination()` yields true, it sets the `self.all_done` `threading.Event()`. The main thread blocked in `main.py` detects this flag, wakes up, sleeps slightly to allow lingering buffer bytes (like final `HAVE` broadcasts) to bleed out smoothly through the TCP stack, and executes termination via `os._exit(0)`, closing the program cleanly without hung daemon threads.
